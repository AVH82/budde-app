# START HERE — Constitution Budd€

Ce document est le point d'entrée obligatoire avant toute modification de Budd€.

## Ordre de lecture obligatoire

1. `docs/START_HERE.md` — comprendre le protocole de contribution.
2. `docs/PROJECT_ID.md` — vérifier l'identité, le périmètre et les contraintes fondamentales du projet.
3. `docs/PROJECT_VISION.md` — comprendre la raison d'être, les valeurs et la trajectoire long terme.
4. `docs/DESIGN_BIBLE.md` — valider la philosophie, l'identité visuelle, l'UX et les règles de cohérence.
5. `docs/TERMINOLOGY.md` — employer le vocabulaire officiel et éviter les termes interdits.
6. `docs/COMPONENT_LIBRARY.md` — réutiliser les composants approuvés avant d'en créer de nouveaux.
7. `docs/ANIMATION_GUIDE.md` — respecter les rythmes, transitions et effets autorisés.
8. `docs/BUDDE_CHARACTER_BIBLE.md` — si Budd€ est concerné, appliquer strictement sa bible personnage.
9. `docs/CODING_GUIDELINES.md` — appliquer les conventions techniques du projet.
10. `docs/AI_COLLABORATION_GUIDE.md` — appliquer la méthode de collaboration IA, d'audit et de PR.
11. `docs/CHANGELOG_DESIGN.md` — vérifier l'historique des validations et y ajouter toute décision graphique validée.

## Règles obligatoires

- Aucune modification ne doit commencer sans lecture de ce document.
- Toute évolution doit respecter simultanément la direction artistique, le vocabulaire, les composants, les animations et les conventions techniques du dossier `docs/`.
- La documentation de `docs/` prime sur les préférences ponctuelles non validées dans le code existant.
- Un nouveau composant, une nouvelle animation, un nouveau terme ou un nouveau comportement de Budd€ doit être documenté dans le fichier de référence correspondant.
- Toute validation graphique durable doit être ajoutée à `docs/CHANGELOG_DESIGN.md`.
- Les données personnelles de l'opérateur ne doivent jamais être ajoutées au dépôt.

## En cas de doute

1. Chercher d'abord la règle dans les documents de référence.
2. Si deux règles semblent contradictoires, appliquer la plus spécifique.
3. Si la contradiction persiste, arrêter le développement et demander confirmation.
4. Si une demande contredit explicitement cette Constitution, ne pas l'implémenter sans confirmation humaine.
5. Si une décision est prise après confirmation, mettre à jour la documentation concernée.

## Définition d'une modification conforme

Une modification est conforme si elle :

- conserve l'identité de terminal rétro-futuriste de Budd€ ;
- reste lisible et ergonomique sur mobile ;
- utilise les termes officiels ;
- réutilise les composants et tokens visuels existants ;
- n'introduit aucune rupture gratuite de style, de rythme ou de comportement ;
- préserve le fonctionnement hors ligne et la simplicité de l'application.
